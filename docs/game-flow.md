# WonderlandUHC 遊戲流程摘要

整理日期：2026-05-21

這份文件摘要說明新版本 WonderlandUHC 從設定、選圖、跑圖、開局到結束的主要流程。目標是讓委託人、原作者與後續維護者快速理解整場 UHC 的生命週期，而不是作為開服安裝教學。

## 核心觀念

1. 遊戲流程分成「世界載入狀態」與「比賽狀態」兩條線。
2. 世界載入狀態負責判斷地圖是否已預覽、是否正在預生成、是否已可開局。
3. 比賽狀態負責等待、傳送、開局倒數與正式遊戲。
4. 選圖完成後會保存 `MatchCenter`，後續邊界、傳送中心與預生成流程都會使用同一個中心。
5. `/uhc choose` 不是直接開始遊戲，而是確認目前預覽世界並進入正式預生成流程。
6. Chunky 是新地圖預生成流程的必要外部插件；缺少時伺服器可啟動，但無法完成正式跑圖流程。

## 世界載入狀態

`LoadingStatus` 用來描述地圖與預生成進度。

| 狀態 | 意義 |
| --- | --- |
| `CONFIGURING` | 尚未產生可用的 UHC 世界，處於設定 / 選圖前狀態。 |
| `WORLD_READY` | 已產生預覽世界，主持人可檢查中心與地形。 |
| `GENERATING` | 主持人已確認預覽世界，伺服器正在進行正式 chunk 預生成。 |
| `DONE` | 預生成完成，世界可用，進入可開局狀態。 |

這個狀態會寫入 cache，因此 `/uhc choose` 後的重啟與 Chunky 預生成完成後的重啟，都能接續上一個流程。

## 比賽狀態

`StateName` 用來描述正式比賽生命週期。

| 狀態 | 意義 |
| --- | --- |
| `WAITING` | 大廳等待與主持設定階段。 |
| `TELEPORTING` | 玩家隊伍正在被分批傳送到 UHC 世界。 |
| `PRE_START` | 玩家已傳送完成，進入正式開局倒數。 |
| `PLAYING` | 遊戲正式開始，PvP、邊界、scenario、死亡與勝負判定進入主要運作階段。 |

世界載入狀態達到 `DONE` 後，玩家才會進入一般等待開局的使用情境；比賽狀態則從 `WAITING` 開始推進。

## 主持流程摘要

### 1. 等待與設定

插件啟動後會載入設定檔、檢查外部插件狀態，並依 cache 判斷目前世界載入狀態。

在 `WAITING` 階段，主持人可調整：

- 隊伍設定。
- 邊界與時間設定。
- scenario。
- 初始物品、死亡掉落物、禁用物品。
- scoreboard 風格與其他遊戲設定。

### 2. 產生預覽世界

主持人使用選單或 `/uhc regen` 產生預覽世界。

流程摘要：

1. 系統詢問是否啟用 CenterCleaner。
2. 若指定 seed，會用該 seed 建立預覽世界。
3. 若啟用 CenterCleaner，系統會在同一張世界中搜尋較適合 UHC 的中心。
4. 搜尋完成後，系統保存目前 `MatchCenter`，並把狀態設為 `WORLD_READY`。
5. 主持人會被傳送到預覽中心，用 creative 模式人工檢查地形。

選圖與分數計算細節見 `docs/map-selection.md`。

### 3. 確認地圖與正式預生成

主持人確認預覽世界後，使用 `/uhc choose`。

這一步會：

1. 保存目前 host、設定與 `MatchCenter`。
2. 將世界載入狀態改為 `GENERATING`。
3. 踢出線上玩家並重啟伺服器。
4. 重啟後接續 cache，透過 Chunky 對 UHC 世界進行 chunk 預生成。
5. 若啟用 Nether 或設定需要強制載入 Nether chunk，會接著處理 Nether。
6. 預生成與邊界處理完成後，將狀態改為 `DONE` 並再次重啟。

因此 `/uhc choose` 的定位是「確認這張地圖並開始正式跑圖」，不是「立刻開局」。

### 4. 開始比賽

當世界載入狀態為 `DONE` 後，主持人可以使用 `/uhc start` 開始大廳倒數。

倒數完成時會：

- 設定初始邊界。
- 依設定處理隊伍分配。
- 關閉玩家正在開啟的選單。
- 清理玩家狀態。
- 讓 staff 傳送到比賽中心。
- 將比賽狀態推進到 `TELEPORTING`。

### 5. 分批傳送

在 `TELEPORTING` 階段，系統會依設定的傳送間隔分批處理隊伍。

流程摘要：

1. 每次處理一個隊伍。
2. 依目前比賽中心與邊界，把隊伍傳送到 UHC 世界中的隨機位置。
3. 傳送後玩家會被凍結，等待正式開局。
4. 所有隊伍傳送完成後，狀態推進到 `PRE_START`。

### 6. 開局倒數

在 `PRE_START` 階段，系統等待正式開局倒數。

倒數完成時會：

- 解除玩家凍結。
- 將參賽玩家設為生存模式。
- 發放初始背包物品。
- 套用初始經驗等級。
- 設定 UHC 世界規則，例如天亮、防火蔓延與難度。
- 發出遊戲開始事件。
- 將比賽狀態推進到 `PLAYING`。

### 7. 正式遊戲

在 `PLAYING` 階段，主要運作包含：

- 傷害開啟、最終回血、PvP 開啟等時間事件。
- 邊界收縮或瞬縮。
- Nether 關閉。
- scenario listener 與各項遊戲規則。
- death handling、spectator 切換與 combat relog。
- 勝負判定。

當系統判定剩下最後勝利隊伍時，會廣播勝利訊息、發出遊戲結束事件，並清除本場世界載入 cache。

### 8. 停止與重置

`/uhc stop` 會刪除世界載入 cache 並關閉伺服器。

這個流程用於結束目前場次或清掉尚未完成的地圖流程。下一次啟動時，若沒有保留可用 cache，插件會回到設定 / 選圖前狀態。

## 實務判斷

1. 如果只是還在挑地圖，停在 `WORLD_READY` 是正常狀態。
2. 如果已按 `/uhc choose`，伺服器重啟並進入 `GENERATING` 是正常流程。
3. 如果 Chunky 沒有安裝或未啟用，正式預生成流程無法完成。
4. 如果世界載入狀態尚未 `DONE`，不要把問題判定為「遊戲不能開始」，而應先確認跑圖流程是否完成。
5. 若比賽已經進入 `TELEPORTING`、`PRE_START` 或 `PLAYING`，再次使用 `/uhc start` 不會重新開始一場新遊戲。

這份摘要只描述主流程。選圖分數、文字格式、外部插件整合與檔案架構分別由其他摘要文件說明。

更完整的狀態、cache、重啟接續、登入限制與開局 timer 細節，見 `docs/details/game-flow.md`。
