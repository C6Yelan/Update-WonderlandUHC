## 權限宣告與程式碼差異

後續狀態（2026-05-21）：`permissions.txt` 已移除，避免它與 `plugin.yml` 重複維護後再次不同步。目前權限宣告以 `plugin.yml` 為準，程式檢查以 `UHCPermission` 與個別 command/menu 檢查為準。以下段落保留為當時盤點與修正歷史。

### `permissions.txt` 落後於 `plugin.yml`

`plugin.yml` 有宣告，但 `permissions.txt` 沒有完整列出的權限：

- `wonderland.uhc.command.giveall`
- `wonderland.uhc.command.practice`
- `wonderland.uhc.command.reconnect`
- `wonderland.uhc.command.uhc.choose`
- `wonderland.uhc.command.uhc.regen`
- `wonderland.uhc.command.uhc.start`
- `wonderland.uhc.host.seebackpack`

### `/setspawn` 權限名稱不一致

程式碼使用：

- `UHCPermission.COMMAND_SET_SPAWN`
- 實際字串為 `wonderland.uhc.command.set.spawn`

但 `plugin.yml` / `permissions.txt` 使用：

- `wonderland.uhc.command.setspawn`

結論：非 OP 若只被授予文件中的 `wonderland.uhc.command.setspawn`，理論上無法通過目前程式碼的 `/setspawn` 權限檢查。

修正狀態（2026-05-20）：已修正，`UHCPermission.COMMAND_SET_SPAWN` 現在明確使用 `wonderland.uhc.command.setspawn`。

### `/respawn` 權限未出現在文件 / plugin.yml 權限宣告

程式碼使用：

- `UHCPermission.COMMAND_RESPAWN`
- 實際字串為 `wonderland.uhc.command.respawn`

但 `plugin.yml` 的 `permissions` 區塊與 `permissions.txt` 沒有列出這個權限。

結論：`/respawn` 指令本身有註冊，但權限文件與 plugin.yml 權限宣告不完整。

修正狀態（2026-05-20）：已修正，`plugin.yml` 與 `permissions.txt` 已補 `wonderland.uhc.command.respawn`，並納入 `moderator` 權限樹。

### `wonderland.uhc.command.uhc.spectoggle` 疑似舊殘留

`plugin.yml` 和 `permissions.txt` 有：

- `wonderland.uhc.command.uhc.spectoggle`

但目前 `spectoggle` 是獨立指令，程式碼檢查的是：

- `wonderland.uhc.command.spectoggle`

結論：`wonderland.uhc.command.uhc.spectoggle` 疑似舊殘留或錯誤權限。

修正狀態（2026-05-20）：已修正，`plugin.yml` 與 `permissions.txt` 已移除此舊殘留，保留 `wonderland.uhc.command.spectoggle`。

### 觀戰聊天權限宣告不完整

原程式碼使用：

- `wonderland.uhc.seespectatorchat`
- `wonderland.uhc.spectator.chat.global`

`permissions.txt` 曾列出兩者；`plugin.yml` 原本只宣告了 `wonderland.uhc.seespectatorchat`，沒有宣告 `wonderland.uhc.spectator.chat.global`。

結論：`wonderland.uhc.spectator.chat.global` 的語意是讓觀戰者全服廣播，與「觀戰者聊天不應被遊玩中玩家看到」的新需求衝突。

修正狀態（2026-05-20）：已決定移除。程式碼、`plugin.yml`、`permissions.txt` 已不再使用 / 宣告 `wonderland.uhc.spectator.chat.global`；保留 `wonderland.uhc.seespectatorchat` 作為管理者查看觀戰聊天的權限。

### 隊伍設定權限

目前 `plugin.yml`：

- `wonderland.uhc.team.settings.name`: `default: true`
- `wonderland.uhc.team.settings.character`: `default: true`
- `wonderland.uhc.team.settings.color`: `default: true`

程式碼 `TeamSettingsMenu` 也會檢查這三個權限。

結論：`/team settings` 入口本身是 `TeamOwnerCommand`，因此最小修正是保留細項權限節點，但把預設值改為 `true`，讓一般玩家可調整自己的隊伍設定。

修正狀態（2026-05-20）：已修正 `plugin.yml` 預設值並通過封裝 / 啟動 smoke test；`_SnYe` 無 OP 實際修改名稱 / 徽章 / 顏色仍待遊戲內回測。

### `/backpack` 權限行為

`plugin.yml` 曾宣告：

- `wonderland.uhc.command.backpack`

但 `BackPackCommand` 沒有檢查 `wonderland.uhc.command.backpack`；它只在查看他人背包時檢查：

- `wonderland.uhc.host.seebackpack`

結論：自己開背包由 Backpack scenario 是否啟用決定；查看他人背包才由 `wonderland.uhc.host.seebackpack` 控制。2026-05-21 已移除未使用的 `wonderland.uhc.command.backpack` 宣告與 `UHCPermission.COMMAND_BACKPACK`，避免誤認為背包功能另有一般指令權限。

## 指令註冊現狀

`plugin.yml` 中宣告的主要指令與目前 `FeatureRegistry.registerNativeCommands()` 註冊流程大致一致。

特別注意：

- `/finish`、`/tohead` 是 `InventoryEditorInputCommand` 的內部輸入指令，沒有額外權限檢查。
- `/whitelist` 是本插件自訂指令，不是原版 `/minecraft:whitelist`。
- 目前指令權限是 command-level 檢查，沒有明確的 host-only 身分檢查；這符合先前測試觀察。
