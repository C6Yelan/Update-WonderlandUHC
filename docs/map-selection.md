# WonderlandUHC 選圖與中心搜尋機制摘要

整理日期：2026-05-21

這份文件摘要說明新版本 WonderlandUHC 的選圖觀念、中心點搜尋方式與分數計算規則。重點是讓委託人、原作者與後續維護者理解：新版不再強制以 `0,0` 作為比賽中心，而是在同一張世界中尋找較適合 UHC 開局與 PvP 的 `MatchCenter`。

## 核心觀念

新版選圖不是追求完美地形，也不是要求世界原點一定要漂亮。

主要觀念：

1. 比賽中心由 `MatchCenter` 表示，包含 X、Z 與初始邊界大小。
2. `0,0` 不再是唯一中心；實際中心會依世界重生點與候選中心搜尋結果決定。
3. 搜尋目標是找到「目前這張世界中相對適合使用的區域」，不是保證每個方向都完全平坦。
4. 地圖選定後，邊界、傳送中心、Nether 對應座標與預生成流程都會使用同一個 `MatchCenter`。
5. 如果主持人不滿意目前預覽結果，可以重新搜尋或指定 seed 重新產生世界。

## 選圖流程摘要

1. 主持人透過選單或 `/uhc regen` 產生預覽世界。
2. 若啟用中心搜尋，系統會以世界重生點作為搜尋基準，建立多個候選中心。
3. 系統逐一抽樣候選中心周圍地形，計算水域、地形、區域分布、中心品質與森林比例。
4. 每個候選中心都會得到分數與狀態。
5. 系統保留分數最高的候選中心，並把它設為本次預覽世界的 `MatchCenter`。
6. 主持人確認後，`/uhc choose` 會進入正式預生成流程；此階段需要 Chunky 才能完成跑圖。

## 不強制 `0,0`

目前做法：

- 搜尋基準是世界重生點，不是固定 `0,0`。
- 候選中心會從搜尋基準往外擴展。
- 最後採用的是評分最高的 `MatchCenter`。
- Overworld 邊界會以 `MatchCenter` 為中心。
- Nether 對應中心會依 Minecraft `1:8` 座標比例換算。

因此，「中心點」代表本場比賽實際使用的中心，不代表世界原點。

## 分數計算摘要

每個候選中心會得到五個主要分數，再加權成總分。

| 項目 | 權重 | 判斷重點 |
| --- | --- | --- |
| 水域 | 22% | 海洋、河流、沼澤與總水域比例越低越好，大片水域過多會扣分。 |
| 地形 | 24% | 高地、極端高地與斷崖比例越低越好，避免中心附近過度崎嶇。 |
| 區域分布 | 18% | 避免單一區域或相鄰區域被大片水域佔據，降低一側地圖品質過差的風險。 |
| 中心品質 | 22% | 檢查中心附近可站立比例、水域比例、高度差與斷崖比例。 |
| 森林 | 14% | 適量森林可接受，但森林或密林比例過高會扣分。 |

總分不是單一條件通過就算好，而是多項因素加權後的結果。這樣可以避免只因中心平坦，就忽略整張邊界內有大量海洋或極端高地。

## 狀態判斷

| 狀態 | 分數區間 | 意義 |
| --- | --- | --- |
| `RECOMMENDED` | `75` 分以上，且沒有主要阻礙原因 | 分數高且沒有主要阻礙，適合作為目前預覽地圖。 |
| `ACCEPTABLE` | `60` 分以上 | 分數達可接受範圍，可使用但品質不一定理想。 |
| `POOR` | `50` 分以上、未滿 `60` 分 | 分數偏低，不建議直接跑圖。 |
| `REJECTED` | 未滿 `50` 分，或命中硬性拒絕原因 | 命中明顯問題，例如水域過高、中心不可站立比例過低或高度差過大。 |
| `TIME_LIMITED` | 不固定 | 搜尋達到時間限制，使用目前已找到的最佳結果。 |
| `CANCELLED` | 不固定 | 搜尋被取消。 |

狀態不是只看分數。若候選中心命中硬性拒絕原因，即使加權總分看起來不低，也會被判定為 `REJECTED`。目前系統會把最佳結果保留給主持人預覽；是否重新搜尋，仍由主持人依實際地形判斷。

## 主要扣分原因

常見扣分原因包含：

- 海洋比例過高。
- 總水域比例過高。
- 單一區域或相鄰區域有大片水域。
- 中心附近大片水域過高。
- 中心可站立比例過低。
- 中心高度差過大。
- 中心斷崖比例過高。
- 低品質區域過多。
- 森林、密林、高地或極端高地比例過高。

這些原因是為了降低 UHC 開局與 PvP 體驗風險，不是為了把所有自然地形清成平地。

## 設定與除錯

目前中心搜尋相關設定在 `settings.yml` 的 `CenterCleaner` 區塊。

| 設定 | 用途 |
| --- | --- |
| `Preview_During_Search` | 搜尋每個候選中心時，是否把主持人傳送到該中心預覽。 |
| `Debug_Search_Output` | 是否輸出每個候選中心的詳細評分資訊。 |
| `Search_Candidate_Count` | 最多評估幾個候選中心。 |

一般使用時不需要開啟詳細除錯。只有在調整選圖規則、追查為什麼某張地圖分數低，或和委託人確認地圖品質時，才建議開啟。

## 實務判斷

這套機制的定位是「輔助主持人快速找到可用地圖」，不是取代人工判斷。

建議判斷方式：

1. `RECOMMENDED` 或 `ACCEPTABLE` 可優先人工預覽。
2. 若中心附近水域、斷崖或森林過多，可重新搜尋。
3. 若整體地形不符合該場需求，可使用 `/uhc regen` 或 `/uhc regen <seed>` 重新產生世界。
4. 確認預覽世界後，再使用 `/uhc choose` 進入正式預生成流程。

更完整的候選中心、抽樣階段、分數門檻、拒絕條件與 debug output 判讀，見 `docs/details/map-selection.md`。
